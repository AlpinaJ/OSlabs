 1339  <p>16.04 runs on systemd. Try the following:</p>
 1340  <pre><code>sudo systemctl restart network-manager.service
 1341  </code></pre>
 1342  <p>If this works, you can create a script to automate it.</p>
 1343  <p>Open a terminal and type the following:</p>
 1344  <p><code>sudo nano /etc/systemd/system/wifi-resume.service</code> Now paste the script in there with a right click. Exit with CTRL + X and press Y to save. Now to activate it: <code>sudo systemctl enable wifi-resume.service</code></p>
 1345  <p>Script:</p>
 1346  <pre><code>#/etc/systemd/system/wifi-resume.service
 1347  #sudo systemctl enable wifi-resume.service
 1348  [Unit]
 1349  Description=Restart networkmanager at resume
 1350  After=suspend.target
 1351  After=hibernate.target
 1352  After=hybrid-sleep.target
 1353  [Service]
 1354  Type=oneshot
 1355  ExecStart=/bin/systemctl restart network-manager.service
 1356  [Install]
 1357  WantedBy=suspend.target
 1358  WantedBy=hibernate.target
 1359  WantedBy=hybrid-sleep.target
 1360  </code></pre>
 1361  <p>Hope this helps. It works on my laptop.</p>

 1371  <p>@147pm Did you ever get this working?</p>
 1372  <p>I found I had a quite similar problem, though I am on Kubuntu 16.10 (KDE-based, not Gnome), and with an HP ProBook laptop. And, unlike yourself, it wasn't my Wifi which died after suspend/wakeup, but my ethernet port.
 1373  Still, I wonder if they are related.</p>
 1374  <p>I also do see that you do not have the problem under KDE. But I would be interested to know if the solution below does help under Gnome, as the solution is not based upon window manager, desktop environment or applets.</p>
 1375  <p>First, just to confirm that restarting the network manager service..</p>
 1376  <pre><code>$ sudo systemctl restart network-manager.service
 1377  </code></pre>
 1378  <p>did not work for me.</p>
 1379  <p>However, I did find an answer that worked, thanks to buzhidao's question and info at <a href="https://askubuntu.com/questions/855699/cant-connect-to-internet-after-suspend">can't connect to internet after suspend</a> and GAD3R's comment there.</p>
 1380  <p>Using their info, I found that first researching which ethernet hardware and driver/module I am running, and then removing and reloading that module, did work for me (though it did not for buzhidao):</p>
 1381  <p>Wifi:</p>
 1382  <pre><code>$ lspci -knn | grep Net -A2
 1383  </code></pre>
 1384  <p>Ethernet:</p>
 1385  <pre><code>$ lspci -knn | grep Ether -A2
 1386  </code></pre>
 1387  <p>The second of these (ethernet) was what I used, and I found:</p>
 1388  <pre><code>03:00.0 Ethernet controller [0200]: Realtek Semiconductor Co., Ltd. RTL8111/8168/8411 PCI Express Gigabit Ethernet Controller [10ec:8168] (rev 0c)
 1389          Subsystem: Hewlett-Packard Company RTL8111/8168/8411 PCI Express Gigabit Ethernet Controller [103c:1944]
 1390          Kernel driver in use: r8169
 1391          Kernel modules: r8169
 1392  </code></pre>
 1393  <p>so i reloaded the 'r8169' driver:</p>
 1394  <pre><code>$ sudo rmmod r8169 &amp;&amp; sudo modprobe r8169
 1395  </code></pre>
 1396  <p>and voila! This worked.  My ethernet port / connection came back alive (after suspend/wakeup) without having to reboot.</p>
 1397  <p>(I also did NOT have a Realtek wifi device, but a Qualcomm Atheros (mod: ath9k) which perhaps explains why wifi continued to work for me after wake-from-suspend.)</p>
 1398  <p>As you can see from my comment on that other post, I wondered whether the problem is the common element between Buzhidao and myself: Realtek Semiconductor devices. Even though they use different modules, they might share some common code? Or even be treated differently by the newer kernel code now in some way?</p>
 1399  <p>Do you yourself have a Realtek-based wifi device? (using lspci above)?  Do you have any luck re-installing the module (rmmod/modprobe above)?</p>
 1400  <p>Anyway, just a shot in the dark. If you have found an eventual answer for yourself, please let us know! Thx.</p>
